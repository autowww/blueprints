Review cadence (suggested)

Reviews are when humans confirm that docs match reality: backlog status, open risks, and release readiness. Tie each row to your change control habits — especially scope updates alongside merged work.

CadenceActivityWhy it matters
Per change request / PRTouch backlog/specs when scope or status changes.Prevents hidden scope: merged code should map to visible work units.
Weekly / end of milestonePlanning/backlog accuracy, open risks, README freshness.Surfaces drift before a release train; good moment for WBS vs board checks.
Before major releaseRelease checklist, privacy/compliance docs, signing process (documented, not secret values).Compliance and store submissions depend on documented process, not tribal knowledge.

If you use engineering-tracking docs under sdlc/, ceremony outputs can draw on the same contributor → events → work units model — see Methodologies & tracking.
