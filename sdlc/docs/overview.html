Overview & roles

This blueprint describes a generic delivery flow and which documentation to update at each step. It pairs with a documentation structure (where files live) and your project’s own requirements or backlog convention.

Path examples often use a docs/ tree; rename folders to match your organization.

How to use this handbook

  • Phases A–F — what happens in each lifecycle phase and which artifacts to touch.
  • Definition of done — when a story or epic counts as complete.
  • Change control — stable IDs, scope management, ADRs, and time tracking (git ≠ duration).
  • Documentation layout — repo shape and milestone → epic → story → task hierarchy.
  • Spec-driven development — intent before code; SDD subchapters define inputs/outputs for C1–C6 and process slots + copy-paste templates.
  • Methodologies & tracking — Scrum, Kanban, XP, phased, Agile, agentic SDLC; links to project sdlc/ when you adopt tracking docs.

Roles

These are accountabilities, not job titles — one person may wear several hats. For archetypes (demand, build, flow, assurance, governance), methodology titles, and how they relate to engineering Contributor identity, see Roles & archetypes (Markdown).

RoleResponsibility
OwnerPrioritizes work breakdown / backlog; accepts epic or story done.
ImplementerImplements work items; updates specs, backlog, and traceability as agreed.
