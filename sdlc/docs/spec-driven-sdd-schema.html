SDD schema & templates

Normative sections and tables for specifying ceremony intents and process slots so inputs, outputs, preconditions, and examples are comparable across teams and safe for agents. Canonical Markdown: SDD-IO-SCHEMA.md.

Purpose

One shape for “what must exist before we start,” “what we consume,” “what we produce,” and “what done looks like”—for both C1–C6 ceremony intents and process slots (refinement, planning, gates, release). Product, engineering, and automation all read the same columns.

YAML frontmatter (normative keys)

---
sdd_id: SDD-C1-ALIGN
kind: ceremony_intent          # ceremony_intent | process_slot
version: "1.0"
status: draft                  # draft | active | deprecated
maps_to:
  ceremony_intents: [C1]
  methodology_events: []
  sdlc_phases: [A, B]
title: "Align on intent — horizon H"
owner_archetype: "Demand & value"
---

Table column shapes

Use one row per artifact, not one row per meeting.

Preconditions

Columns: ID · Artifact · Format · Owner · Ready when

Inputs

Columns: ID · Name · Type · Source · Format · Example (snippet)

Outputs

Columns: ID · Name · Type · Primary consumer · Storage / system · Example (snippet)

Quality rules

  1. Examples are mandatory — at least one filled example per spec.
  2. Stable IDs — change sdd_id or version when semantics shift.
  3. Every output has a consumer — person, role, or system.
  4. Chat is not storage — name repo path, ticket, or tool.
  5. Map to C1–C6 where the slot is ceremony-shaped — keeps alignment with ceremony foundation.
