CI/CD, quality gates, and test plans

CI/CD — continuous integration (build + checks on push/PR) and, when adopted, delivery/deployment automation.

ConceptWhat to capture
PipelineWhere it lives (e.g. .github/workflows/), which branches trigger it.
Quality gatesMandatory steps: compile, lint/static analysis, unit tests, optional scans — document so “green” matches team expectations.
Test planFor meaningful scope: what to test (levels, environments), manual/exploratory ownership, exit criteria — story spec or templates/TEST-PLAN.template.md.
TraceabilityTests ↔ requirements in a matrix or inline in code, per project convention.

Merge (suggested): No merge to main while required CI checks fail, unless Owner accepts risk and notes a waiver in the PR/story.

Release (suggested): No release artifact until the same (or stricter) gates pass on the release commit/tag.

Details belong in docs/development/ — not in the frozen blueprint prose beyond SDLC.md §7.

Optional — agents & automation: Repositories may add a sibling blueprints/agents/ package for containerized automation recipes (shell, browser runners, optional tool loops) alongside IDE/LLM workflows governed elsewhere. See the handbook chapter Agents & automation and canonical blueprints/agents/STRUCTURE.md.
