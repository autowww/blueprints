Agents & automation

Software agents here means any automation that behaves like a teammate: **LLM assistants**, **IDE / editor tools**, **CI bots**, **codegen/test generators**, and **containerized** recipes. Humans still own intent and acceptance; see Agentic SDLC. The optional **blueprints/agents/** package is the repeatable execution layer (Docker images, Compose, agents/recipes) — not the only way to use agents, but the supported pattern for isolated, auditable jobs. Canonical Markdown: blueprints/agents/STRUCTURE.md.

Relationship to this handbook

Process and repo-wide docs/ layout stay in blueprints/sdlc/. Agentic practices (how work is proposed, reviewed, and traced) are methodology — Agentic SDLC. This chapter is about where automation runs: optional blueprints/agents/ defines how to run repeatable jobs in containers; it does not replace CI/CD & quality gates — it implements optional gate steps (e.g. browser/E2E, scripted checks) when you adopt it.

What a solid agent setup looks like

Regardless of vendor, align the repo so agents cannot bypass governance:

  • Intent in the repo — backlog / REQ ids, specs, or ADRs; chat is not the system of record.
  • Reviewable output — PRs, commits, CI; label bot accounts if your policy requires it.
  • Repeatable heavy tasks — move long-running or environment-sensitive work into agents/recipes (or CI jobs) so local and pipeline match.
  • Optional isolation — blueprints/agents/ supplies base images and Compose so recipes do not depend on whoever installed what on a laptop.

Purpose

  • Isolation — scripts and tools run in Linux containers; host drift affects outcomes less.
  • Repeatability — same image + documented mounts → similar behavior locally and in CI.
  • Separation — frozen patterns in blueprints/agents/; mutable jobs in agents/ at repo root.

Docker specifics

  • Base image — default Node 20 (Debian slim) + git, curl, jq. Extend or add a second image for Playwright (system deps or mcr.microsoft.com/playwright).
  • Mounts — bind repository to /work (read-only when possible); writable outputs to agents/workspaces/ on the host.
  • Compose — build context is blueprints/agents/; see docker/compose.yaml.
  • Resources — for Chromium/Playwright, increase shared memory (shm_size) in CI when needed.

DeterminismPin tool versions in the recipe (lockfiles, config).

Security

Non-root users where possible; read-only root filesystem when supported; restrict outbound network in CI if recipes allow; pin base image digests for sensitive pipelines. Full table: STRUCTURE.md §6.

CI integration

Typical pattern: CI job builds agents-blueprint-base:local (or pulls a cached image), then docker compose run a recipe with repo and workspace mounts. List required agent jobs beside compile/test in docs/development/ so “green” matches team expectations (CI/CD chapter).

Bootstrap agents/ & new recipes

Workspace: ./blueprints/agents/scripts/init-agents-workspace.sh — creates agents/ with README, workspaces/, and recipes/.

New recipe: ./blueprints/agents/scripts/new-agent-recipe.sh <name> [--purpose "…"] — scaffold under agents/recipes/<name>/.

Canonical playbook: blueprints/agents/ORCHESTRATION.md · scripts/README.md.

Evolution

Single project: one base image + a few shell recipes. Growing: split images per stack (Node vs Playwright vs Python); use Compose profiles. Monorepo: mount package subpaths or set per-recipe WORKDIR.
