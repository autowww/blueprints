IDE agent instructions

How the SDLC methodology is projected into the native instruction formats of AI coding agents (Cursor, Claude Code). Templates for rules, skills, commands, root instruction files, and a plan convention — bootstrapped by init-ide-workspace.sh and customized per project. Canonical Markdown: blueprints/ide/STRUCTURE.md.

Relationship to other chapters

Agents & automation covers Docker containers and CI recipes (blueprints/agents/). Agentic SDLC covers governance (humans own intent, transparency, review). This chapter covers IDE agent configuration: how the process is encoded so AI assistants can follow it. Spec-driven development is the methodology that underpins the instruction content.

Why IDE agent instructions?

AI coding agents (Cursor, Claude Code) work best when they receive persistent, actionable context at the start of every session. Without it, agents must rediscover your workflow, conventions, and expectations on each conversation — or worse, they guess.

The blueprints/ide/ package solves this by projecting SDLC methodology (spec-driven development, agentic governance, commit conventions) into the formats each vendor tool consumes natively: .cursor/rules/, CLAUDE.md, commands, skills, and plan templates.

What gets generated

Running ./blueprints/ide/scripts/init-ide-workspace.sh "Project Name" copies templates into the project, replacing {{PROJECT_NAME}} and stripping the .template. segment from filenames.

Root instruction files

  • AGENTS.md — loaded by Cursor at session start. Contains build/test commands, spec-driven workflow, commit conventions, review contract.
  • CLAUDE.md — same role for Claude Code, with session hygiene guidance and @-imports.

Cursor rules (.cursor/rules/)

  • spec-driven.mdc — work-unit IDs, acceptance criteria, spec lifecycle. Activated on docs/requirements/**/* globs.
  • agentic-workflow.mdc — governance: human intent, transparency, review. Description-activated.
  • commit-conventions.mdc — work-unit refs, message format, Co-authored-by. Always applied.
  • plan-and-verify.mdc — plan-first workflow, verification contract, context hygiene. Description-activated.

Cursor commands (.cursor/commands/)

  • /plan — create a plan artifact linked to a work-unit ID.
  • /pr — commit with work-unit ref, push, create PR.
  • /review-spec — compare code against spec acceptance criteria.

Claude Code skills & agents

  • spec-driven skill — auto-activated or /spec-driven.
  • plan-and-verify skill — auto-activated or /plan-and-verify.
  • spec-reviewer subagent — reviews changes against spec criteria in a separate context.

Plan convention

A plan is a short-lived markdown artifact bridging the spec (durable intent) and the implementation (code). It is not a durable spec — it guides a single implementation cycle.

When to create a plan

  • Multi-file changes (3+ files or multiple system boundaries)
  • Unfamiliar code areas
  • Complex logic, data migrations, security-sensitive changes
  • Uncertain approach with multiple valid strategies

When to skip

  • Single-file fixes, config tweaks, log additions
  • Well-known patterns already in the codebase
  • Trivial changes describable in one sentence

Lifecycle

Create (plan mode) → Review → Use (during implementation) → Discard or archive (after merge).

Plans are named <work-unit-id>-<short-title>.md and stored in .cursor/plans/ (typically gitignored) or docs/plans/ (if git-tracked).

Context & session hygiene

AI agent performance degrades as the context window fills with irrelevant information. The generated instructions encode these practices:

  • One work unit per conversation when practical — switching tasks mid-conversation dilutes context.
  • Start fresh when the agent seems confused, keeps making the same mistake, or effectiveness decreases.
  • Delegate investigation to subagents (Claude Code) or separate tasks (Cursor) so exploration doesn't fill the main context.
  • Plan mode first for large changes — separates research from execution.

Adoption

  1. Ensure blueprints/ide/ is present (submodule or copy).
  2. Run: ./blueprints/ide/scripts/init-ide-workspace.sh "Project Name"
  3. Fill in project-specific sections: build/test/lint commands, project decisions.
  4. Commit the generated files; add .cursor/plans/ to .gitignore if plans should stay local.
  5. Use --cursor-only or --claude-only if the team uses only one tool.
  6. Use --force to refresh from newer templates (diff before committing).

Customization after bootstrap

Generated files are mutable project files, not blueprint text. Teams should:

  • Adjust rules for their stack, naming conventions, and preferred tools.
  • Add project-specific rules alongside blueprint-derived ones.
  • Fill in the "Project decisions" section in AGENTS.md / CLAUDE.md.
  • Add more commands or skills as workflows emerge.

If improvements are generic, consider backporting them into blueprints/ide/templates/ so the next project gets the same baseline.

Anti-patterns

Anti-patternFix
Over-specified instructions (agent ignores them)Keep rules under 500 lines; split into composable files. If the agent already does it correctly, delete the rule.
Cursor rules and CLAUDE.md drift apartMaintain one set of canonical content in templates; both vendor files derive from the same source.
Trusting without verificationAlways provide tests, linters, or scripted checks. If you can't verify it, don't ship it.
Agent as Product OwnerHuman review for acceptance, scope, security, data, public API.
Plans never discardedPlans are session artifacts. Archive or delete after merge; don't maintain them long-term.
