Methodologies & tracking

How Scrum, Kanban, phased delivery, and XP use one engineering-tracking foundation — plus where the Markdown sources live in the consuming repo.

Project tracking files under the consuming repo’s sdlc/ (e.g. TRACKING-METHODOLOGIES.md) stay lightweight. Full methodology guides (Scrum, Kanban, …) live in blueprints/sdlc/methodologies/ — canonical Markdown + handbook pages below.

Cross-cutting scope management and time tracking (git activity vs explicit hours) are in the Change control chapter — methodology docs below assume you align backlog and work-unit ids with those rules.

Document hierarchy (three files)

One spine for identity and events; methodologies as lenses; challenges documented separately so limits stay visible.

  • TRACKING-FOUNDATION.md — stable model only.
  • TRACKING-METHODOLOGIES.md — full methodology tables and summary.
  • TRACKING-CHALLENGES.md — what is hard or misleading.

Methodology lenses (on the same foundation)

Each row configures time windows, “done” for reporting, and ceremonies — without a second data model.

LensLayers on foundationTypical extra inputs
ScrumSprint window, sprint backlog linkage, review/retro aggregatesSprint dates, backlog membership, capacity from ALM
KanbanContinuous or rolling windows; flow-oriented reportsBoard columns, blockers, created→done from tracker
PhasedPhase labels on work units, milestone windows, phase touchPhase taxonomy, REQ map, compliance/ALM for gates
XPSmall-batch signals, pairing (Co-authored-by:), quality correlationCI dashboards, pairing norms

Agile (umbrella): combine Scrum or Kanban cadence with XP practices — still one foundation; see the Markdown for nuance.
