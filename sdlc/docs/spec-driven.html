What we mean

A spec is any durable artifact that defines what must be true when work is complete: story files, epic briefs, acceptance tests (examples or automated), API contracts, ADRs, and checklists your team treats as binding. Spec-driven development is the discipline of having those artifacts ready and agreed before—or in tight iteration with—implementation, so “done” is provable, not subjective.

Why it matters

OutcomeHow specs help
Shared understandingOwner, implementers, and reviewers align on done before deep coding.
TraceabilityStable work-unit IDs link commits, tests, and releases to explicit intent.
Less reworkAmbiguity surfaces in cheap refinement, not after merge.
Agentic workflowsLLMs and CI bots need durable context in the repo; chat is not the system of record — see Agentic SDLC.
Regulated deliveryEvidence chains (req → design → test → release) assume specs exist and change through control.

Minimal artifact set

Adjust paths to your docs/ tree; the pattern is what counts.

ImplementTasks, PRs; code comments for why, not duplicate specsImplementers
VerifyTests tied to criteria; manual checklist if neededImplementers + Owner
ReleaseRelease notes referencing story / requirement IDsTeam

Rule: If nobody can point to a spec sentence or test that justifies a change, either the work is speculative or the spec must be updated first.

Lifecycle loop (docs ↔ code)

  1. Draft — Intent in milestone/epic/story; assign IDs early (Change control).
  2. Refine — Acceptance criteria testable and agreed (Definition of Ready).
  3. Implement — Small batches; PRs trace to a work unit.
  4. Inspect — Review checks spec fit; CI checks executable specs where they exist.
  5. Adapt — If reality invalidates the spec, update prose or add an ADR in the same change set as code when possible.

This aligns with Phases A–F: discovery and specification stay visible in the repository.

Specs and agents

Agents amplify execution; humans still own acceptance and prioritization. Effective practice: keep stories, criteria, and IDs under docs/requirements/ (or equivalent) before large automated edits; require human review for security, data, and public API changes; treat failing tests or unmet criteria as the stop signal. Chapter Agents & automation covers the optional execution layer.

Anti-patterns

Anti-patternFix
Code-first, spec-laterBlock done until criteria and traceability match what shipped.
Specs only in chatMove decisions into Markdown (or exported tracker) in the tree.
Vague prosePrefer examples, given/when/then, or checklists.
Stale specsPR discipline: behavior change without spec update is scope drift.
One giant documentUse milestone → epic → story hierarchy so diffs stay reviewable.
