Handbook

Product functional documentation

What the product does for users and stakeholders — capabilities, journeys, and behavioral rules — in one coherent structure.

Source of truth: Markdown in blueprints/docs/ (STRUCTURE.md, README.md). When those change, update this HTML and MAINTENANCE.md.

Overview

This blueprint complements delivery process (blueprints/sdlc/) and backlog / requirement specs (e.g. docs/requirements/). It defines where to write product-functional documentation: vision, personas, capability map, journeys, feature behavior, data and integrations at a product level.

Mutable content in the consuming repo typically lives under docs/product/. Copy templates/ from this blueprint to seed files — do not put project prose inside blueprints/docs/.

Principles

PrincipleMeaning
Audience-firstReadable without the backlog tool.
Just enoughAdd docs when they reduce ambiguity about observable behavior.
Single source of truthLink to requirement IDs; avoid duplicating full acceptance criteria.
Journeys before deep specsPrimary flows first; detailed feature specs second.
TraceabilityOptional stable IDs (CAP-…, JRN-…, FEAT-…) linking to backlog IDs.

Folder layout (mutable)

Typical tree under docs/product/:

docs/product/
  INDEX.md
  vision/
  personas/
  capabilities/
  journeys/
  features/
  data/
  integrations/
  glossary.md
  assumptions-open-questions.md

Rename product/ if your organization prefers functional/ or prd/.

Document types

TypeWhen to add / update
Vision & scopeDirection, goals, in/out of scope change.
Personas / segmentsICP or constraints change.
Capability mapMajor abilities appear, merge, or retire.
User journeysCore flows change.
Feature / domain specsBehavior, rules, errors, edge cases need clarity.
Data (product view)Privacy, retention, conceptual model at product level.
Integrations (product view)New systems or changed contracts visible to users.
Glossary & assumptionsDomain terms; explicit uncertainty.

Relationship to other documentation

LayerRole
blueprints/docs/Frozen IA + templates (this package).
docs/product/Living product functional content.
docs/requirements/Backlog, milestones, story specs — link by ID.
docs/architecture/Technical structure; functional docs describe behavior, not implementation.
Root README.mdSetup and high-level features — keep short.

Governance

The blueprints/docs/ directory is governed by POLICY.md: do not edit casually. Project-specific functional documentation belongs in docs/product/.

See also README.md and POLICY.md in the parent folder.
